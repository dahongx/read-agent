# 文献阅读 Agent 版本规划

**文档日期：** 2026-04-15  
**当前版本：** v1.0  
**当前定位：** 单篇论文阅读与汇报生成 MVP  

---

## 一、项目目标

本项目的目标不是只做一个“论文转 PPT”工具，而是逐步发展为一个面向学术汇报场景的智能文献助手。它应当帮助用户完成以下整条链路：

1. 获取论文
2. 理解论文
3. 生成汇报材料
4. 辅助口头讲解
5. 支持现场问答
6. 在后续版本中支持多论文对比、自动检索与知识积累

从当前阶段来看，系统已经完成了“单篇论文上传后自动生成 PPT + 讲稿 + 问答”的基础闭环，但还没有完全达到 PRD 中“多篇文献数据库式使用”的目标。因此，后续版本规划应当围绕“先把单篇做稳，再扩展到多篇，再进入自动找论文”这一主线推进。

---

## 二、PRD 对照与当前实现情况

结合 [PRD.md](/E:/agent/reading%20agent/PRD.md) 和当前代码实现，项目当前完成情况可以归纳如下。

### 1. 已经实现的部分

- 支持上传单篇 PDF，并完成文本解析、分块、索引和后续处理。
- 支持根据论文内容自动生成 PPT，前端可直接预览，并可导出下载。
- 支持同步生成逐页讲稿，并在页面中进行语音朗读。
- 支持左侧 PPT、右侧问答的双栏展示方式，符合汇报场景。
- 支持围绕当前论文进行知识库问答，并提供页码级引用与原文跳转。
- 支持语音输入提问、语音播报答案，基础演示流程已经跑通。

### 2. 目前还没有实现的部分

- 暂不支持一次上传多篇论文。
- 暂不支持基于关键词自动检索论文。
- 暂不支持输入论文 URL 后自动抓取 PDF。
- 暂不支持多篇论文联合问答与对比分析。
- 暂不支持将历史上传论文沉淀为可持续复用的个人文献库。

### 3. 当前版本的准确定位

当前 v1.0 更准确的描述应当是：

**一个面向单篇论文场景的智能汇报生成系统，而不是完整的文献数据库系统。**

这一定义很重要，因为它决定了下一阶段的工作不应该一上来做“大而全”，而是要先把单篇场景的稳定性、可用性和展示效果打磨好，再向多篇和自动检索扩展。

---

## 三、v1.0 已交付能力总结

### 1. 单篇论文处理闭环已经形成

当前系统已经能够完成以下完整流程：

`上传 PDF → 解析论文 → 生成 PPT → 生成讲稿 → 页面展示 → 文内问答 → 原文定位`

这意味着项目已经具备一个可演示、可汇报、可继续迭代的 1.0 基础版本。

### 2. 从产品角度看，v1.0 的价值已经成立

对于老师布置的“先做出一个能用的版本”这一目标来说，v1.0 已经满足基本要求：

- 用户不需要手动总结论文。
- 用户可以快速拿到一套汇报材料。
- 用户在展示过程中可以边讲边问。
- 用户面对追问时，系统可以给出答案并回溯原文位置。

### 3. v1.0 当前更像 MVP，而不是终版

目前版本更适合定义为：

**MVP（Minimum Viable Product，最小可用产品）**

它证明了产品方向是成立的，但还存在三个明显限制：

- 只能处理单篇论文，无法满足综述、对比、组会常见场景。
- 论文来源仍依赖用户手动上传，自动化程度不够。
- 问答链路已可用，但在多轮对话、精确检索、稳定性方面仍有提升空间。

## 四、版本演进总览

后续版本建议按以下顺序推进：

| 版本 | 核心目标 | 关键词 |
|------|------|------|
| v1.0 | 单篇论文基础闭环跑通 | 上传、生成、问答、引用 |
| v1.1 | 单篇论文稳定版 | 可用性、容错、前端交互优化 |
| v1.2 | 多论文联合处理版 | 多篇上传、联合问答、综述生成 |
| v1.3 | 自动找论文版 | 关键词检索、URL 导入、自动下载 |
| v2.0 | 个人学术助手版 | 文献库、跨论文检索、长期积累 |

这个节奏的好处是清晰、可落地，也符合你目前“1.0 先这样，下一步继续优化”的实际推进方式。

---

## 五、v1.1 规划：单篇稳定版

v1.1 的目标不是增加太多新功能，而是把单篇论文场景真正做到“稳定可用、可交付、可给别人试用”。

### 1. 重点优化方向

- 优化上传与生成流程，减少前端在配置和提交阶段的报错。
- 增强生成状态提示，让用户明确知道当前处于“解析中、生成中、导出中”的哪一步。
- 优化失败重试机制，避免一个环节报错后整个流程直接中断。
- 明确缓存命中提示，让用户知道是“重新生成”还是“读取已有结果”。
- 改善生成结果页的可用性，例如缩略图导航、翻页体验、讲稿展示层次等。

### 2. 建议补齐的具体能力

- 上传前的参数校验与默认值兜底。
- 生成过程中的进度展示与错误提示统一化。
- 讲稿区支持复制、导出、局部编辑。
- 问答区支持保留最近几轮历史。
- 结果页支持重新生成同一论文但更换模板或页数。

### 3. 为什么 v1.1 要先做这些

因为当前项目已经能“做出来”，但还需要先解决“稳不稳、顺不顺、别人能不能直接上手”的问题。只有单篇链路足够稳定，后面扩展到多篇时才不会把复杂度一起放大。

---

## 六、v1.2 规划：多论文联合处理版

这是后续最关键的一个版本，因为它决定项目能否真正从“单篇工具”升级为“文献分析平台”。

### 1. 核心目标

支持一次处理多篇论文，并围绕多篇论文开展问答、对比和汇报生成。

### 2. 建议实现的能力

- 支持批量上传多个 PDF。
- 每篇论文分别生成自己的 PPT 与索引结果。
- 支持多篇论文联合问答。
- 支持回答时标注来源论文标题与页码，而不只是页码。
- 支持生成“多篇论文综述 PPT”，而不是只生成单篇解读 PPT。

### 3. 用户层面的典型场景

- 输入 3 篇同方向论文，系统生成一份对比型汇报。
- 询问“这几篇论文的方法差异是什么”，系统能跨论文回答。
- 询问“哪篇论文实验最好”，系统能给出对比结论并标出处。

### 4. 技术上会带来的变化

这个版本会影响现有的 Session、索引和任务管理设计：

- Session 不再对应单个 PDF，而是对应一个论文集合。
- 检索不再是单索引，而是多索引或合并索引。
- 引用信息不再只是 `(第N页)`，而应升级为“论文名 + 页码”。
- PPT 生成可能分为“两种模式”：单篇分别生成，或多篇合成综述。

因此，v1.2 很可能是第一个需要明显调整后端数据结构的版本。

---

## 七、v1.3 规划：自动找论文版

在多论文上传能力打通之后，下一步就应该把“找论文”也纳入系统。

### 1. 核心目标

让用户不再必须先手动下载 PDF，而是可以直接输入研究主题、关键词或论文链接，让系统自动完成检索、下载和后续生成。

### 2. 建议支持的三种入口

#### 1. 本地上传

保留现有方式，适合用户已经有 PDF 文件时直接使用。

#### 2. 关键词搜索

用户输入研究方向或关键词，例如：

- `long context language model`
- `memory augmented llm`
- `retrieval augmented generation`

系统自动搜索论文列表，展示标题、作者、摘要、年份，供用户勾选后继续生成。

#### 3. URL 导入

用户粘贴论文页面地址，系统自动抓取 PDF 并继续处理。

优先建议支持：

- arXiv
- ACL Anthology
- Semantic Scholar
- PubMed

### 3. 这一版本的意义

如果 v1.2 解决的是“多篇处理”，那么 v1.3 解决的就是“论文来源自动化”。这一步完成后，产品体验会从“你先准备好论文，我帮你处理”，升级为“你只告诉我研究主题，我帮你把后续流程跑下去”。

## 八、问答系统优化路线

当前问答已经可以使用，但如果后面要继续优化体验，建议不要只盯着模型本身，而要把重点放在“检索质量、上下文组织、答案可信度”上。

### 1. 多轮对话上下文

当前问答更像单轮问答，后续应补上最近几轮历史，使系统能理解“它、这个方法、上一页那个实验”等追问。

### 2. 混合检索

当前如果主要依赖向量检索，遇到模型名、指标名、公式名、数字型问题时会不够稳。建议加入 BM25 或关键词检索，与向量检索做融合。

### 3. 重排序

对初筛出的检索片段再做 rerank，减少相关性不高但被错误召回的片段进入最终回答。

### 4. 章节感知检索

后续可以让系统识别论文结构，例如：

- Abstract
- Introduction
- Method
- Experiment
- Conclusion

这样用户可以限定“只在方法部分回答”或“优先参考实验部分”。

### 5. 短论文全文模式

对于篇幅较短的论文，不一定要严格走分块检索，可以尝试全文放入上下文，减少切块带来的信息割裂。

### 6. 多论文分层检索

在多篇论文场景中，建议采用“两层检索”：

1. 先判断最相关的是哪几篇论文
2. 再在这些论文内部检索具体段落

这样比直接在所有 chunk 里混搜更稳。

### 7. 回答可信度与引用约束

建议后续增强以下机制：

- 检索证据不足时，允许系统明确回答“论文中未找到明确依据”。
- 回答内容尽量绑定引用，不引用就尽量不下结论。
- 引用页码和实际证据内容要做一致性检查，避免“看起来有出处，实际对不上”的问题。

## 九、无服务器情况下的分享方案

你前面提到后续要把项目发给别人使用，如果没有正式服务器，也可以有几种过渡方案。

### 1. 本地启动 + 内网访问

适合在同一个局域网内演示，例如实验室、教室、办公室。

- 你本地运行前后端服务。
- 其他人通过你的局域网 IP 和端口访问。
- 优点是最省事。
- 缺点是只能同网络下访问，且你的电脑必须保持运行。

### 2. 本地启动 + 隧道工具临时暴露公网

适合临时给别人一个可访问链接。

可选方式：

- ngrok
- Cloudflare Tunnel
- frp

优点：

- 不需要自己买服务器。
- 可以快速得到一个公网链接给别人访问。

缺点：

- 更适合演示和短期试用，不适合长期正式使用。
- 稳定性、速度、并发能力都有限。
- 如果后端依赖本地模型、CLI 或较复杂环境，迁移和分享时容易出兼容问题。

### 3. 静态前端单独托管

如果后面把一部分能力改成“前端静态页面 + 远程 API”，则前端可以单独部署到：

- Vercel
- Netlify
- GitHub Pages

但目前这个项目前后端耦合较强，且生成和问答都依赖后端，所以暂时不适合作为完整方案。

### 4. 现阶段的建议

如果只是近期给老师、同学、朋友试用，建议先采用：

**本地部署 + 隧道工具分享链接**

这是当前成本最低、改动最小的办法。等后续版本稳定后，再考虑正式服务器或云部署。

## 十、优先级建议

结合当前项目阶段，下一步建议按照以下顺序推进：

### P0：优先马上做

- 整理和稳定单篇上传到生成的完整流程。
- 修复前端上传和参数交互问题。
- 补充问答多轮上下文。
- 明确分享方案，先支持本地启动后通过隧道临时外发。

### P1：下一阶段重点做

- 多篇论文上传与联合问答。
- 回答中加入论文名 + 页码级引用。
- 混合检索与 reranking。
- 生成综述型 PPT。

### P2：版本能力增强

- 关键词检索论文。
- URL 自动抓取论文。
- 章节感知检索。
- 讲稿编辑与导出。

### P3：更长期再考虑

- 个人文献库沉淀。
- 历史会话持久化。
- 团队协作共享。
- 对外 API 服务。

---

## 十一、阶段性结论

当前项目的 1.0 版本已经完成了一个清晰的最小闭环：

**单篇论文上传 → 自动生成 PPT 与讲稿 → 页面展示 → 文内问答 → 原文定位**

这说明项目方向是成立的，产品核心价值已经验证完成。

下一步不建议立刻把所有 PRD 能力一次性做完，而是建议分三步推进：

1. 先把单篇流程稳定下来，形成真正可用的 v1.1。
2. 再扩展到多篇论文联合处理，完成产品能力升级。
3. 最后补上关键词检索、URL 导入和自动下载，让系统具备更完整的自动化能力。

按照这个节奏推进，项目路线会更清晰，开发风险也更可控，后续无论是继续自己优化，还是交给别人接手，都会更容易说明白。
